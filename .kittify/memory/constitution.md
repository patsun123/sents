## Purpose

This constitution captures the technical standards, code quality expectations,
tribal knowledge, and governance rules for the Sentiment Stock Exchange (SSE).
All features and pull requests must align with these principles.

SSE is a solo project. Architecture and tooling choices prioritize simplicity,
automation, and resilience over complexity. Any human or AI agent should be

## Technical Standards

### Languages and Frameworks

| Layer | Technology |
|-------|-----------|
| Backend | Python 3.12+ with FastAPI (async) |
| Frontend | TypeScript with React 18 + Vite |
| Database | PostgreSQL (persistent storage) |
| Cache | Redis (real-time prices, sentiment scores) |
| Sentiment | VADER / Transformers (Python NLP ecosystem) |
| Reddit | PRAW (Reddit API client) |
| Finance | Alpaca or yfinance (market data) |
| Infra | Docker + Docker Compose |

### Testing Requirements

- **Backend**: pytest, minimum 90% line coverage, 100% for critical paths (sentiment engine, price calculation)
- **Frontend**: Vitest + React Testing Library, minimum 85% coverage
- **E2E**: Playwright for critical user flows (price display, divergence visualization)
- **CI**: All tests must pass before any merge — CI is the sole gatekeeper
- **Mocking**: Reddit and finance API calls must be mocked in unit/integration tests
- **No human QA** — automated test coverage is the primary quality assurance mechanism

### Performance and Scale Targets

- 10,000 concurrent users
- 1,000,000 daily active users
- API response time: p95 < 200ms
- Sentiment pipeline latency: < 5 seconds from Reddit post ingestion to price update
- Redis caching is mandatory (not optional) to meet these targets

### Deployment and Constraints

- **Local**: Docker Compose for development and testing
- **Production**: Cloud-agnostic Docker containers — deploy to Fly.io, Railway, or Render
- **No vendor lock-in**: All services containerized and portable
- **Resilience**: Auto-restart policies and health checks on all services
- **Degradation**: Graceful degradation when Reddit or finance APIs are unavailable
- **Observability**: Structured logging throughout + Sentry for error alerting (no 24/7 monitoring required)
- **Solo constraint**: The system must be self-healing — alerts come to the developer, not the other way around

## Code Quality

### Pull Request Requirements

- Solo project — self-merge is allowed after CI passes
- CI is the mandatory gatekeeper: all checks must be green before merge
- No exceptions to CI requirements

### Code Review Checklist

Before merging, verify:

- [ ] Tests added or updated for all changed behavior
- [ ] Docstrings updated for all public functions, classes, and modules
- [ ] Code follows PEP 8 (backend) and project TypeScript conventions (frontend)
- [ ] No security issues introduced (injection, auth bypass, secrets in code, etc.)
- [ ] Type annotations present on all functions (Python + TypeScript)
- [ ] Error handling is robust — no silent failures, no bare `except` clauses
- [ ] Performance implications considered (N+1 queries, missing cache usage, etc.)
- [ ] Code is correct, clear, and maintainable

### Quality Gates

All of the following must pass before merge:

1. All tests pass (pytest + Vitest + Playwright)
2. Coverage ≥ 90% backend, ≥ 85% frontend
3. Linter clean (ruff for Python, ESLint for TypeScript)
4. Type checker clean (mypy for Python, tsc for TypeScript)
5. Security scan clean (bandit for Python, npm audit for frontend)

### Documentation Standards

- **All public APIs**: docstrings with description, parameters, return values, and usage examples
- **README**: kept current — updated whenever a new feature is added or setup changes
- **ADRs**: Architectural Decision Records written for any significant architectural choice
- **Inline comments**: required for non-obvious logic; code should explain *what*, comments explain *why*
- **OpenAPI**: auto-generated from FastAPI — always reflects the live API state
- **Goal**: Any human or AI agent should be able to understand the system fully without asking questions

### Lessons Learned

**Keep it simple. Do not over-engineer the solution.**

- Solve the problem in front of you, not the hypothetical future problem
- Reach for the simpler tool first — add complexity only when the simpler approach provably fails
- Every dependency is a liability; add only what is necessary
- A working simple system beats a broken complex one
